## Context

当前系统采用单事件识别设计，`IntentResult` 和 `deepSeekUnifiedIntent` 结构仅包含单个事件字段（`event_name`、`action`、`event_id`）。用户输入如"没吃，睡着了"包含两个独立意图（结束喂养+开始睡眠），但系统只能识别其中一个事件。

**当前数据流转：**
```
用户输入 → 向量匹配/LLM → IntentResult(单事件) → Go落库(单记录)
```

**限制：**
- 一次只能处理一个喂养事件
- 复合意图（多事件）无法正确识别和处理
- 用户需要多次输入才能完成多个事件的记录

## Goals / Non-Goals

**Goals:**

1. 支持用户输入中包含多个喂养事件的识别（如"没吃，睡着了"→结束喂养+开始睡眠）
2. 扩展意图分析响应结构，支持返回多事件列表
3. Go 侧支持一次处理多个事件并批量落库
4. 保持向后兼容，单事件场景无需修改调用方
5. 数据飞轮支持多事件的用户表达记录

**Non-Goals:**

1. 不支持嵌套事件（事件中包含子事件）
2. 不支持超过 3 个事件的复合意图（防止过度复杂）
3. 不修改向量匹配逻辑（向量匹配仍返回单事件，LLM 负责多事件识别）
4. 不支持部分确认（用户确认即确认所有事件，取消即取消所有事件）

## Decisions

### Decision 1: IntentResult 结构扩展

**设计：** 在 `IntentResult` 中增加 `events` 字段（列表类型），每个元素包含 `event_name`、`action`、`event_id`。

**理由：**
- 向后兼容：单事件场景仍使用原有字段，多事件场景使用 `events` 列表
- 清晰分离：单事件和多事件数据结构明确区分
- Go 侧易于解析：只需遍历 `events` 列表即可处理多个事件

**数据结构：**
```python
# Python 侧 IntentResult
{
    "target_type": "feeding",
    "action": "multi",  # 标记为多事件
    "event_name": "",   # 单事件时使用
    "event_id": "",     # 单事件时使用
    "events": [         # 多事件列表
        {"action": "end", "event_name": "母乳喂养", "event_id": "1"},
        {"action": "start", "event_name": "睡眠", "event_id": "2"},
    ],
    "keywords": [],
    "content": "",
}
```

### Decision 2: LLM 提示词扩展

**设计：** 修改意图分类提示词，引导 LLM 识别复合意图，支持返回多事件列表。

**理由：**
- LLM 具备识别复合意图的能力，只需适当引导
- 向量匹配仍返回单事件，多事件识别由 LLM 完成
- 提示词中明确说明支持多事件的 JSON 返回格式

**提示词修改：**
```
意图类型说明增加：
- feeding: 喂养记录相关意图
  - multi: 多个喂养记录（如"没吃，睡着了"包含结束喂养和开始睡眠）

返回格式增加：
{
    "target_type": "feeding",
    "action": "multi",
    "events": [
        {"action": "end", "event_name": "事件名", "event_id": "ID"},
        {"action": "start", "event_name": "事件名", "event_id": "ID"}
    ]
}
```

### Decision 3: 确认流程设计

**设计：** 多事件场景下，`prepare_confirm` 生成统一的确认消息，用户一次确认/取消所有事件。

**理由：**
- 简化用户交互：一次确认即可完成所有事件记录
- 实现简单：无需复杂的部分确认逻辑
- 数据一致性：要么全部确认，要么全部取消

**确认消息示例：**
```
您是要：结束记录「母乳喂养」，开始记录「睡眠」吗？请回复「确认」或「取消」。
```

### Decision 4: 数据飞轮设计

**设计：** 用户确认后，将完整用户表达（如"没吃，睡着了"）关联到第一个事件，后续向量匹配时可直接匹配到完整表达。

**理由：**
- 保持数据飞轮的简单性
- 用户表达与事件的关联是一对一的（一个表达对应一个事件列表）
- 后续相同输入可直接向量匹配成功

### Decision 5: Go 侧处理逻辑

**设计：** 修改 `handleUnifiedIntentAction`，当 `action == "multi"` 时遍历 `events` 列表，逐个落库。

**理由：**
- 复用现有落库逻辑，只需增加循环处理
- 保持事务一致性：所有事件都落库成功或都失败
- 向后兼容：单事件场景仍使用原有逻辑

## Risks / Trade-offs

### Risk 1: LLM 可能返回过多事件

**Mitigation:** 在提示词中限制最多返回 3 个事件，超过时只取前 3 个。

### Risk 2: 部分事件落库失败

**Mitigation:** Go 侧使用事务批量落库，任一事件失败则全部回滚。

### Risk 3: 确认消息过长

**Mitigation:** 事件超过 2 个时，简化确认消息格式（如"您确认要记录这些事件吗？"）。

### Risk 4: 向量匹配与多事件的冲突

**Mitigation:** 向量匹配仍返回单事件，多事件识别由 LLM 完成。当向量匹配置信度<0.9 时走 LLM 分析。

### Trade-off: 向后兼容性 vs 功能完整

**Decision:** 保持向后兼容，单事件场景使用原有字段，多事件场景使用 `events` 列表。Go 侧优先检查 `events` 字段，为空时使用原有字段。
