## Context

当前系统中，喂养意图识别使用简单的关键词包含判断（`strings.Contains`），缺乏语义理解能力，容易产生误判。例如"奶粉喂养知识"可能被误判为"奶粉喂养"记录事件。同时，喂养动作和喂养建议的代码混合在同一目录中，难以快速定位和维护。

**当前架构**：
- Python 服务作为"智能内核"，被 go_ai_talk 调用
- 使用 Chroma 向量数据库存储母婴知识文档
- 使用 cachetools.TTLCache 缓存事件字典（24小时TTL）
- LangGraph 用于编排意图分析和诊疗流程

**约束条件**：
- Python 服务必须通过 HTTP 调用兄弟仓 API，不能直接访问数据库
- 事件字典缓存必须使用 24 小时 TTL
- 向量数据库 volume 必须在运行时挂载，首次启动时构建
- 所有代码必须包含详细中文业务逻辑注释

## Goals / Non-Goals

**Goals:**

1. 引入向量数据库进行语义相似度匹配，提升喂养意图识别准确性
2. 实现置信度分层判定：≥95%直接判定，90%-95%需要确认，<90%走LLM
3. 使用 LangGraph interrupt 实现用户确认机制
4. 实现数据飞轮：用户确认后将表达添加到向量库
5. 错误处理：置信度≥90%但用户否定时删除向量数据
6. 事件字典自动同步：24小时更新时同步向量库
7. 代码模块化重构：分离喂养动作和喂养建议代码
8. 制定全局代码注释规范

**Non-Goals:**

1. 不考虑服务重启后的中断状态恢复
2. 不引入新的外部依赖（如 Redis 持久化）
3. 不修改 go_ai_talk 代码（由用户手动实现）
4. 不改变现有 API 接口结构（仅添加新字段）

## Decisions

### 决策 1：向量库结构设计

**选择**：使用独立的 `feeding_events` Collection，与 `mother_baby_knowledge` 物理隔离

**理由**：
- 避免事件数据和知识数据互相干扰
- 便于独立管理和清理
- 支持不同的检索策略

**替代方案考虑**：
- 在同一 Collection 中使用 metadata 区分 → 检索时需过滤，增加复杂度

### 决策 2：置信度分层规则

**选择**：
- ≥95%：直接判定，跳过用户确认
- 90%-95%：需要用户确认，用户否定时删除向量数据
- <90%：走 LLM 分类，需要用户确认

**理由**：
- 高置信度场景（≥95%）用户体验优先，跳过确认
- 中等置信度场景（90%-95%）需要确认，错误时删除向量避免重复
- 低置信度场景（<90%）依赖 LLM 进行最终判定

**替代方案考虑**：
- 所有场景都确认 → 用户体验差
- 仅向量匹配需要确认 → LLM 解析错误无法纠正

### 决策 3：用户确认实现方式

**选择**：使用 LangGraph interrupt 机制

**理由**：
- 标准做法，支持图执行暂停和恢复
- 与现有 LangGraph 架构一致
- 无需额外状态管理

**替代方案考虑**：
- 手动状态管理 → 增加复杂度，容易出错

### 决策 4：数据飞轮触发条件

**选择**：仅当用户确认 LLM 解析的意图正确时，才添加用户表达

**理由**：
- 保证向量库数据质量
- 避免错误表达污染向量库

**替代方案考虑**：
- 所有确认都添加 → 可能引入低质量数据

### 决策 5：错误处理策略

**选择**：置信度≥90%但用户否定时，直接删除对应的向量数据

**理由**：
- 避免高置信度错误重复出现
- 保持向量库精简，避免膨胀

**替代方案考虑**：
- 记录负样本 → 向量库膨胀，影响检索性能

### 决策 6：事件字典同步机制

**选择**：在事件字典缓存更新时（24小时TTL过期），自动检测变化并同步向量库

**理由**：
- 利用现有缓存机制，无需额外调度
- 确保向量库与事件字典保持一致

**替代方案考虑**：
- 定时任务同步 → 增加复杂度，可能与缓存更新冲突

### 决策 7：代码组织方式

**选择**：将喂养动作和喂养建议代码分离到独立文件夹

```
app/
├── feeding/           # 喂养动作
│   ├── graphs/
│   ├── schemas/
│   └── services/
├── clinic/            # 喂养建议
│   ├── graphs/
│   ├── schemas/
│   └── services/
├── config/            # 共享配置
└── shared/            # 共享服务
```

**理由**：
- 便于问题追溯和维护
- 模块边界清晰
- 支持独立测试和部署

**替代方案考虑**：
- 保持现有结构 → 代码混杂，难以维护

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 向量库膨胀 | 检索性能下降 | 设置最大记录数（10,000条），定期清理低质量样本 |
| 误删标准事件名 | 匹配能力下降 | 删除时检查 `source != "standard"` |
| 用户体验波动 | 低置信度场景需要多次确认 | 数据飞轮持续提升匹配质量 |
| 服务启动延迟 | 首次启动时构建向量库 | Docker 构建阶段预构建知识向量库 |
| 中断状态丢失 | 用户确认流程中断 | 不考虑服务重启后的状态恢复 |
| 向量匹配失败 | 无法识别用户意图 | 回退到 LLM 分类 |

## Migration Plan

### 部署步骤

1. **Phase 1：代码重构**
   - 创建新目录结构（`app/feeding/`、`app/clinic/`、`app/shared/`）
   - 迁移现有代码到新结构
   - 制定并应用代码注释规范

2. **Phase 2：向量库增强**
   - 修改 `vector_store.py`，新增 `feeding_events` Collection
   - 修改 `event_cache.py`，支持向量库同步
   - 扩展 `build_vector_db.py`，支持喂养事件向量库构建

3. **Phase 3：意图分析增强**
   - 修改 `intent_graph.py`，添加向量匹配节点
   - 添加用户确认机制（interrupt）
   - 实现数据飞轮和错误处理逻辑

4. **Phase 4：部署配置**
   - 修改 `Dockerfile`，添加向量库自动构建步骤
   - 更新 `docker-compose.yml`，添加向量数据库 volume 挂载

### 回滚策略

- 如果新代码出现问题，回退到原代码结构
- 向量数据库使用 volume 挂载，回滚时保留数据
- API 接口保持向后兼容，不影响 go_ai_talk 调用

## Open Questions

1. **向量库容量阈值**：最大记录数设置为多少合适？当前建议 10,000 条
2. **清理算法细节**：低质量样本的评估指标和清理比例需要进一步确定
3. **标准事件动作变体**：是否需要为所有标准事件添加"开始/结束/记录"动作变体
4. **事件层级追问**：如何处理父事件有子节点时的追问逻辑
